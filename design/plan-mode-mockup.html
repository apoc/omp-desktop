Fix duplicate streaming tool-call writes

overview

The reducer is appending a fresh ToolCallState on every streaming delta instead of upserting against the call's stable id. Two writers (tool_call_started and tool_call_finalized) race the same id; both fall through the append branch because the lookup keys mismatch — finalized uses callId, started writes id.

approach

  1. Normalise the lookup key once at the boundary (RPC adapter), not in the reducer.
  2. Change the reducer's tool-call branch to upsert by id, removing the redundant pendingToolCalls queue.
  3. Route tool_call_finalized through the active map; preserve the existing ToolCallState shape so the renderer doesn't flicker.

key steps

  • adapter — collapse id/callId to a single canonical field at parse time
  • reducer — replace append with upsertById in the streaming branch
  • store/session — drop pendingToolCalls; assert it's empty in the migration test
  • tests — add a regression test that fires started + 3× delta + finalized and asserts a single entry
auto-advances to review when streaming stops · or abort to discard draft and edit intent
